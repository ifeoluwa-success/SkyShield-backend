## 1. OVERVIEW

**System purpose**
- SkyShield Edu is an aviation cybersecurity simulation backend built with Django, Django REST Framework, Django Channels, PostgreSQL, and JWT auth.
- It supports mission-based cyber range training for airport personnel through scenario simulations, real-time meetings, analytics, and legacy LMS-style content.

**Core architecture**
- urls.py exposes API namespaces:
  - `/api/auth/`
  - `/api/users/`
  - `/api/content/`
  - `/api/tutor/`
  - `/api/simulations/` (includes scenarios, sessions, structured **courses**, enrollments, course certificates, incidents, genie)
  - `/api/analytics/`
  - `/api/core/`
  - `/api/meetings/`
- WebSockets are mounted separately in ASGI:
  - `/ws/meeting/<room_name>/`

**Authentication method**
- JWT authentication using `djangorestframework-simplejwt`.
- Custom login endpoint returns JWT access + refresh tokens.
- Social login supports Google and GitHub via `dj_rest_auth` and `allauth`.
- Email verification and password reset are implemented with token models.

**Role system**
- User roles defined in models.py:
  - `trainee`
  - `supervisor`
  - `admin`
  - `instructor`
- Role enforcement exists in:
  - meeting host checks
  - tutor/trainer filtering
  - custom view behavior
- There is no explicit API endpoint in code for assigning roles; role changes appear managed outside these API routes.

**Real-time components**
- The meeting real-time channel is under `/ws/meeting/<room_name>/`.
- Real-time behavior supports meeting join/waiting-room, WebRTC signaling, chat, media state, and admission controls.
- A separate mission real-time channel exists for simulation incident runs:
  - `/ws/mission/<run_id>/`

#### `/api/simulations/sessions/`

- Auth: required

#### `/api/simulations/sessions/start/`
- Method: `POST`
- Description: Start a new simulation session for a scenario.
- Request body:
  - `scenario_id` (UUID, required)
- Behavior:
  - Rejects if scenario inactive or max attempts exceeded.
  - Reuses existing in-progress session if present.
  - Creates `SimulationSession` with `status='in_progress'`, `session_state=scenario.initial_state`.
  - Creates `UserActivity(activity_type='simulation_start')`.
- Response: serialized `SimulationSession`.

#### `/api/simulations/sessions/submit_decision/`
- Method: `POST`
- Description: Submit a decision for the current simulation step.
- Request body:
  - `session_id` (UUID)
  - `step_number` (int)
  - `decision_type` (choice)
  - `decision_data` (JSON)
  - `time_taken` (int)
- Behavior:
  - Validates active session and current step.
  - Determines correctness via `correct_actions` in the current scenario step.
  - Persists `UserDecision`.
  - Updates session metrics:
    - `total_choices`
    - `correct_choices`
    - `time_spent`
    - adds mistake details to `session_state['mistakes']` if wrong
    - updates `session_state` via `update_state`
    - recalculates `accuracy_rate`
  - If more steps remain, increments `current_step` and returns `next_step`.
  - If final step reached:
    - Sets `status='completed'`
    - Sets `completed_at`
    - Calls `calculate_score()`
    - Updates user level stats and scenario stats
    - Creates achievements
    - Calls `AdaptiveLearningService.update_user_profile(user, session)` if available
    - **Course pipeline:** when `session.status == 'completed'`, calls `CourseService.record_simulation_result(session.id, request.user)` so linked course modules can advance; failures are logged and never block the completion response.
- Response:
  - if continuing:
    ```json
    {
      "correct": true,
      "feedback": { ... },
      "next_step": { "number": 1, "title": "...", "description": "...", "options": [...] },
      "session": { ... }
    }
    ```
  - if completed:
    ```json
    {
      "completed": true,
      "score": 85.5,
      "passed": true,
      "feedback": { ... },
      "summary": { ... },
      "session": { ... }
    }
    ```
- Errors:
  - `404 Active session not found`
  - `400 Simulation already completed`
  - `500 Invalid step data`

##### Scoring formula
- `base_score = (correct_choices / max(total_choices, 1)) * 100`
- `time_penalty = max(0, (time_spent - scenario.estimated_time*60)/60) * 0.5`
- `hint_penalty = hints_used * 2`
- `final_score = max(0, base_score - time_penalty - hint_penalty)`
- `passed = score >= scenario.passing_score`

##### Accuracy formula
- `accuracy_rate = (correct_choices / total_choices) * 100`

## 5. REAL-TIME / WEBSOCKET APIs

### WebSocket URL
- `/ws/meeting/<room_name>/`
- `/ws/mission/<run_id>/`

### Connection auth
- Auth via query string token:
  - `?token=<JWT access token>`
- The backend uses `rest_framework_simplejwt.tokens.AccessToken` to resolve the user.
- Invalid or missing token closes the connection with code `4001`.

### Connection behavior
- On connect:
  - validates token
  - resolves meeting by `room_name`
  - fetches current user role
  - if meeting missing: close code `4004`
  - if participant record cannot be created: close code `4003`
  - if waiting room is enabled and user is a `trainee`, places them into waiting room
  - if admitted, adds them to meeting group and broadcasts join events

### Incoming message types
- `offer`
  - fields: `type`, `target`, `offer`
- `answer`
  - fields: `type`, `target`, `answer`
- `ice_candidate`
  - fields: `type`, `target`, `candidate`
- `chat`
  - fields: `type`, content, optional `recipient_id`, optional `message_type`
- `admit`
  - fields: `type`, `participant_id`
- `get_participants`
  - fields: `type`
- `media_state`
  - fields: `type`, `videoEnabled`, `audioEnabled`
- `screen_share`
  - fields: `type`, `sharing`

### Outgoing event formats
- `room_info`
  - sent on successful connection
  - payload includes participants and meeting metadata
- `waiting`
  - sent to waiting-room trainees
- `user_joined`
- `user_left`
- `admitted`
- `waiting_room_update`
- `participant_list`
- `chat`
- `media_state`
- `screen_share`
- `offer`
- `answer`
- `ice_candidate`

### Real-time mission channels
- Simulations define a mission real-time channel:
  - Group name: `mission_{run_id}`
  - Auth: `?token=<JWT access token>`
  - Close codes:
    - `4001` invalid/missing token
    - `4004` run not found
    - `4003` user not a mission participant

### Mission WebSocket incoming message types
- `submit_action`
- `acknowledge_briefing`
- `request_hint`
- `abandon`
- `supervisor_intervention`
- `get_state`

### Mission WebSocket outgoing message types
- `state_snapshot` (sent on connect and on `get_state`)
- `mission_event` (broadcast to group)
- `state_update` (broadcast full snapshot to group)
- `timer_warning` (broadcast when 15 seconds remain)

- `participant_list`
- `chat`
- `media_state`

### Event payload structure
- `offer` / `answer` / `ice_candidate`
  - `type`
  - `target`
  - `offer` / `answer` / `candidate`
- `chat`
  - `type: "chat"`
  - content
  - optionally `recipient_id`
  - optionally `message_type`
- `admit`
  - `type: "admit"`
  - `participant_id`
- `media_state`
  - `type: "media_state"`
  - `videoEnabled`
  - `audioEnabled`
- `screen_share`
  - `type: "screen_share"`
  - `sharing`

### Lifecycle flow
1. client connects with `token` query string
2. backend validates user and meeting
3. user enters live room or waiting room
4. backend sends `room_info` or `waiting`
5. connect triggers `user_joined` broadcast for live users
6. host uses `admit` to move waiting trainees into room
7. offer/answer/ice_candidate relay peer signaling
8. chat/media events are broadcast group-wide

### Missing event types
- Meeting WebSocket does not use these names, but the mission engine persists them as `IncidentEvent.event_type` values:
  - `action_submitted`
  - `phase_changed`
  - `escalation_triggered`
  - `hint_requested`
  - `intervention_applied`
  - `participant_joined`
  - `participant_left`
  - `timeout_occurred`
  - `genie_event`
  - `system`

## 13. SYSTEM BEHAVIOR RULES

### Simulation state updates
- State updates are stored in `SimulationSession.session_state`.
- `update_state()` merges `step_data['state_updates']` and tracks visited steps.
- Each decision persists as `UserDecision`.

### Event sourcing rules
- There is no explicit event sourcing layer or replay timeline API.
- Session history is reconstructed from `SimulationSession` and `UserDecision` items.

### Replay logic assumptions
- `history` endpoint is the only replay-like access.
- No dedicated timeline reconstruction endpoint exists.

### Concurrency behavior
- User sessions and simulation sessions are scoped by `request.user`.
- No explicit locking or transaction manager is visible beyond standard Django ORM operations.
- Meeting WebSocket state is persisted into `MeetingParticipant` and broadcast through channel groups.

### Role-based access constraints
- Authenticated access is required for nearly all API routes.
- Some endpoints use `permissions.IsAdminUser`.
- Meeting host checks are enforced in meeting actions.
- Trainee-specific logic exists in tutor and meeting flows.
- No dedicated role assignment API route exists.
